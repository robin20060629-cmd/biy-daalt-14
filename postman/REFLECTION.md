### 1. Аль assertion хамгийн их үнэ цэнэтэй санагдсан вэ? Яагаад?
**Schema + property шалгалт** хамгийн их үнэ цэнэтэй байлаа. `pm.expect(d).to.have.property('id')` гэх мэт шалгалтууд нь API-ийн "contract"-г тодорхойлдог. 

Status 200 ирсэн ч response body-д шаардлагатай талбар байхгүй байж болно. Жишээлбэл, `price` талбар алга болсон бол frontend тооцоолол хийж чадахгүй гацах эрсдэлтэй. Энэхүү assertion нь API хувилбар өөрчлөгдөхөд гардаг "breaking change"-ийг тэр даруй илрүүлдэг давуу талтай. Ялангуяа `pm.expect(d.email).to.match(/@/)` гэх мэт бизнес дүрмийн шалгалт нь зөвхөн талбар байгааг биш, утга нь логиктой эсэхийг баталгаажуулдаг тул маш хүчтэй шалгалт болдог.

### 2. Negative test жишээгээ дэлгэрэнгүй тайлбарла
**"POST Login — Буруу нууц үг"** тест:
* **Оролт:** `username: "wronguser"`, `password: "wrongpassword"`
* **Шалгадаг зүйл:** Сервер 400 эсвэл 401 алдаа буцааж байгаа эсэх, мөн `accessToken` ирээгүй эсэхийг баталгаажуулна.
* **Илрүүлдэг алдаа:** Хэрэв сервер алдаатай нэвтрэх мэдээлэлд 200 (OK) + токен буцаавал энэ нь аюулгүй байдлын ноцтой алдаа юм. 
* **Ач холбогдол:** `pm.expect(d).to.not.have.property('accessToken')` гэсэн шалгалт нь алдааны зам (error path) зөв ажиллаж байгааг баталгаажуулдаг. Алдааны үед систем хэрхэн хариу үзүүлэх нь "хэвийн ажиллах" замаас ч дутуугүй чухал гэрээ (contract) юм.

### 3. Postman-д амжилттай ажиллаж байсан тест Newman-д fail болсон уу? Яагаад?
Тийм. **GET Auth/Me — Token байхгүй** тест Postman дээр "Pass" болсон боловч Newman дээр анх ажиллуулахад "Fail" болсон.

* **Шалтгаан:** Postman-ийн GUI дээр ажиллах үед өмнөх хүсэлтээс үлдсэн `{{token}}` environment variable локал кэшд хадгалагдсан байсан тул "No Auth" гэж тохируулсан ч Authorization header-ийг автоматаар нэмж байсан. Харин Newman-д environment файл нь цэвэр (placeholder-той) байсан тул токенгүй хандахад 401 алдаа ирж, тест "Fail" болсон.
* **Шийдвэр:** Request-ийн тохиргоо дээр `auth: { type: "noauth" }` гэсэн override-ыг зөв хийснээр хоёр орчинд ижил үр дүнтэй болгосон.
